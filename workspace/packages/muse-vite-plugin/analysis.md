# Shared Module Registration: Problem Analysis and Solutions

## Background

MUSE lib plugins (e.g. `muse-lib-react`) bundle React and other shared dependencies and expose them
through a global registry so that normal plugins can consume them without duplicating the dependency.
The runtime API is:

```js
// Provider (lib plugin) — called at load time
MUSE_GLOBAL.__shared__.register(modules, __require__)

// Consumer (normal plugin) — called at runtime
MUSE_GLOBAL.__shared__.require("react@18.2.0/index.js")
```

The build pipeline (`museRolldownPlugin`) must inject registration calls into each lib plugin bundle
so that every module the lib exposes is registered before any normal plugin loads.

---

## Problem Description

### Registry internals

`register(modules, __require__)` in `muse-modules/register.js` stores:

```js
MUSE_GLOBAL.__shared__.modules[mid] = { id: mid, __require__ };
```

`require(museId)` in `muse-modules/museRequire.js` later calls:

```js
return m.__require__(m.id);
```

The `__require__` factory is the only path back to the actual module value. If the factory returns
an empty object, the consumer gets nothing.

### Boot ordering guarantee

`muse-boot-default/src/boot.js` (lines 250–271) loads all lib plugin bundles via a single
`<script type="module">` element and `await`s a Promise that resolves only after the entire import
chain settles (`window.__onLibEntriesResolve()`). Normal plugins are loaded only after this
completes. So any registration that finishes before that Promise resolves — including microtasks
scheduled within the module graph — is guaranteed to be complete before any consumer calls
`__shared__.require()`.

### Build configuration

Lib plugin output is configured as:

```js
format: 'es'
codeSplitting: false    // single-file bundle
treeshake: false
```

All modules are flattened into one `main.js`. Static ESM `import` statements in source are
compiled by Rolldown into internal variable references within that file.

## Solution 1 — Static Self-Import (`import * as mmmmm from self`)

### Implementation

The `transform` hook appends two lines to **every** module in the lib plugin:

```js
import * as mmmmm from "/absolute/path/to/this-module.js";
MUSE_GLOBAL.__shared__.register({ "react@18.2.0/index.js": mmmmm }, () => mmmmm);
```

### How it was expected to work

Static `import * as mmmmm` is hoisted by the ES module spec before the module body runs.
For a self-import, `mmmmm` is the module's own namespace object — a live binding proxy.
The factory `() => mmmmm` captures this live reference. Since `register` stores the factory
and only calls it lazily (when a consumer calls `require()`), and by that time all modules are
fully evaluated, `mmmmm` should reflect the complete namespace.

### Why it fails

In Rolldown's **bundled** single-file ESM output the self-import is compiled by the
`__exportAll` runtime helper:

```js
// generated by Rolldown at the top of the module section — before any exports are assigned
var react_exports = /* @__PURE__ */ __exportAll({
  default: () => react_default,
  useState: () => useState,
});

// ... actual variable assignments ...
var useState = ...;
var react_default = ...;

// registration — mmmmm was renamed to react_exports
MUSE_GLOBAL.__shared__.register({ "react@18.2.0/index.js": react_exports }, () => react_exports);
```

`__exportAll` creates the namespace object **immediately** (at the top of the module section)
using getter functions that close over the not-yet-assigned variables. Depending on Rolldown's
version and output mode, the getters may or may not reflect the variables' final values by the
time the `register()` call executes synchronously. In practice this produces an **empty or
partially-initialized namespace** at registration time, causing consumers to receive `{}` or
`undefined` exports.

Additionally, static `import` statements are hoisted to the top of the module by the JS parser
regardless of where they appear in source, so appending `import * as mmmmm from self` at the
bottom of the module code does not help — the namespace object is still created before the module
body begins executing.

### Pros

- Synchronous — no async machinery needed.
- Simple to generate in the `transform` hook.
- Produces self-contained per-module registration code.

### Cons

- **Circular self-import** — every module imports itself, creating a cycle that is fragile
  across different Rolldown versions and output formats.
- **Namespace populated too early** — the namespace object captured by `() => mmmmm` may be
  empty because it is created before the module's own `let`/`const` bindings are initialized.
- **Per-module code injection** — every single source file gets extra lines appended, inflating
  the bundle and making the generated output harder to read.
- **Brittle** — correctness depends on undocumented Rolldown behaviour around circular imports
  and `__exportAll` getter evaluation timing.

## Solution 2 — Dynamic Import (`.then()`)

### Implementation

The `transform` hook appends a dynamic import instead of a static one:

```js
import(/* "/absolute/path/to/this-module.js" */).then(m => {
  MUSE_GLOBAL.__shared__.register({ "react@18.2.0/index.js": m }, () => m);
});
```

### How it works

`import()` returns a Promise that resolves only after the target module has **finished
evaluating**. By the time `.then(m => ...)` fires, `m` is the fully-populated namespace — all
`let`/`const`/`var` bindings, including re-exported ones, are initialised. The factory `() => m`
captures this complete snapshot.

Because `.then()` is a microtask, and because `boot.js` awaits the entire `<script type="module">`
execution including all microtasks scheduled within it before loading normal plugins, the
registration is guaranteed to be complete before any consumer calls `__shared__.require()`.

### Why this was replaced

The team wanted registration to be **synchronous** from the developer's mental model — no
Promise chains, no timing dependency, no concern about microtask scheduling. The async approach
also makes it harder to reason about errors (a rejected `.then()` is a silent unhandled
rejection unless explicitly caught).

### Pros

- **Correct** — module is fully evaluated before the factory is stored.
- **No circular import** — `import(self)` is a dynamic import; module systems typically handle
  this by returning the already-evaluated module from cache rather than re-executing it.
- **Simple to generate** — one extra expression per module in the `transform` hook.

### Cons

- **Async** — registration happens in a microtask, not synchronously during module evaluation.
  This introduces a conceptual ordering dependency on the boot sequence.
- **Per-module code injection** — same verbosity issue as Solution 1.
- **Silent failure risk** — if the dynamic import fails for any reason the `.then()` callback
  never runs and the module is silently unregistered.
- **Harder to test** — asynchronous registration complicates unit tests that check registry state.

## Solution 3 — Virtual Entry + Register Module (Implemented)

### Implementation

Two virtual modules replace the per-module code injection:

**`\0muse-virtual-entry`** (new build entry for lib plugins):

```js
import "/absolute/path/to/actual/src/index.js";
import "\0muse-shared-register";
```

**`\0muse-shared-register`** (generated in the `load` hook after all modules are collected):

```js
import * as __muse_m0__ from "/path/to/react/index.js";
import * as __muse_m1__ from "/path/to/react-dom/index.js";
import * as __muse_m2__ from "/path/to/scheduler/index.js";
// ...one line per lib module...

const __muse_modules__ = {
  "react@18.2.0/index.js":     __muse_m0__,
  "react-dom@18.2.0/index.js": __muse_m1__,
  "scheduler@0.23.0/index.js": __muse_m2__,
};
MUSE_GLOBAL.__shared__.register(__muse_modules__, (id) => __muse_modules__[id]);
```

The `transform` hook retains the `sharedModules[mid] = id` collection but injects no code.

### Why the timing is safe

Rolldown builds module graphs **depth-first** following static import declaration order. The
virtual entry lists `actualEntry` before `\0muse-shared-register`. Therefore Rolldown fully
traverses the entire actual-entry dependency tree — running `transform` on every module and
populating `sharedModules` — before it calls the `load` hook for `\0muse-shared-register`.
By the time the register module's code is generated, `sharedModules` is complete.

### Why the namespace is populated

The imports in `\0muse-shared-register` are **not circular** — the register module is a
distinct module that imports each lib module. In Rolldown's bundled single-file output, these
compile to:

```js
// each lib module is already flattened into the bundle; __muse_mN__ is an alias to its exports var
var __muse_modules__ = {
  "react@18.2.0/index.js":     react_exports,
  "react-dom@18.2.0/index.js": react_dom_exports,
};
MUSE_GLOBAL.__shared__.register(__muse_modules__, (id) => __muse_modules__[id]);
```

Because this code is appended **after** all the actual lib module sections in the flat bundle,
all export variables (`react_exports`, `react_dom_exports`, …) are fully assigned before the
`register()` call executes. The factory `(id) => __muse_modules__[id]` then returns the correct,
fully-populated namespace synchronously.

### Pros

- **Correct and synchronous** — no Promises, no microtasks, no circular imports.
- **No per-module code injection** — source modules are not modified; the `transform` hook
  only reads module IDs.
- **Single registration point** — one `register()` call covers all modules, making the generated
  bundle easy to inspect and debug.
- **Non-circular** — `\0muse-shared-register` imports lib modules; lib modules do not import
  `\0muse-shared-register`. Evaluation order is unambiguous.
- **Namespace fully initialized** — by the time the register module's code runs in the flat
  bundle, every lib module's variables are assigned.

### Cons

- **Depth-first ordering dependency** — correctness relies on Rolldown processing the virtual
  entry's first import (`actualEntry`) before its second import (`\0muse-shared-register`).
  This is standard bundler behaviour and holds in Rolldown (as in Rollup), but is not an
  explicit API guarantee.
- **More moving parts** — requires `resolveId`, `load`, and a changed build `input` in addition
  to the existing `transform` collection logic.
- **Entry replacement** — lib plugin build input changes from the actual entry file to
  `\0muse-virtual-entry`; any tool that reads `build.rolldownOptions.input` directly would see
  the virtual ID instead of a real file path.

## Comparison Table

| Criterion | Static self-import | Dynamic import | Virtual entry |
|---|---|---|---|
| Synchronous | Yes | No (microtask) | Yes |
| Circular import | Yes (self) | No | No |
| Namespace fully initialized | No (fragile) | Yes | Yes |
| Per-module code injection | Yes | Yes | No |
| Relies on boot ordering | No | Yes | No |
| Bundle readability | Poor | Poor | Good |
| Implementation complexity | Low | Low | Medium |
| **Status** | **Broken** | **Working** | **Implemented** |
